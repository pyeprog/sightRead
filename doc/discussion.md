# 设计讨论：从导航历史到代码阅读的指引

> 日期：2026-07-06
> 性质：第一性原理层面的定位讨论，非具体实现方案

## 一、插件的初心

这个插件是为**阅读重量级项目**设计的。

最初的痛点：顺着定义树或引用树往下走，走到一定深度就会**迷失**。想回去，但不一定能干净地回到某个地方——Ctrl+O、Ctrl+I 来回按很多下才能回到想去的位置。

所以最初只想做一件**简单且稳固**的事：把走过的代码位置可视化地记录下来，随时能一步跳回之前看过的地方。

## 二、时代背景的变化：Vibe Coding

现在情况不同了：大部分代码由 Vibe Coding 产生。当你需要开始了解、阅读这样一个项目时，出现了新的困难：

1. **不知道从哪开始读**——对项目可能一无所知，找不到起点。
2. **代码量巨大，不知道重点在哪**——无法分辨哪些是骨架、哪些是细节。
3. **读不懂时求助 LLM，但 LLM 的输出和代码是脱节的**——它只能输出文本和链接，只能靠肉眼把解释和代码对应起来。

## 三、抽象出的三个需求

人类的理解需要**指引**。面对一个项目，阅读行为可以抽象为三个层面的需求：

### 需求 1：找入口

一个项目通常有若干个入口。有些入口可以直接执行，有些不能被用户简单执行，而是通过某种方式被其他程序调用。找到这些入口是阅读的起点，是一个普遍需求。

### 需求 2：看路径

从入口开始自上而下走，看到挂在这棵树下的**每一个节点提供的功能是什么**，从而：

- 了解整体架构，或者
- 定位某个具体功能在哪个节点里。

关键在于此时要**略过海量细节**，从整体的大架构上去 construct structure。

### 需求 3：理解一个具体的实现

范围通常是一个函数。需要理解这个函数的内部结构，例如：

- 这一段代码是 corner case 的判断，然后提前退出；
- 这一段代码是主要算法；
- ……

## 四、分析（Claude 的观点）

### 4.1 三个需求 = 同一件事的三个尺度

三个需求本质上是"理解"这一活动在三个尺度上的展开：

| 需求 | 尺度 | 类比 |
|------|------|------|
| 找入口 | 项目尺度的**定位** | 一本书的目录 / 一座城的城门 |
| 看路径 | 架构尺度的**结构** | 章节结构 / 主干道地图 |
| 理解实现 | 代码尺度的**细读** | 逐段精读 / 街景 |

人的工作记忆装不下一个重量级项目，所以理解必然依赖一张**外部化的、多尺度的地图**。这个框架比"导航历史"这个原始定位更接近问题的本质。

### 4.2 关键观察：足迹与地图是一对对偶

原始功能和三个新需求不是同一类东西，而是互补的对偶：

|  | 原始功能（导航历史） | 三个新需求（指引） |
|--|--------------------|-------------------|
| 方向 | 自下而上 | 自上而下 |
| 性质 | **足迹**：记录你实际走过的地方 | **导游图**：告诉你该往哪走 |
| 来源 | 你自己的阅读行为 | LLM / 静态分析 |
| 覆盖面 | 窄（只有去过的地方） | 广 |
| 可信度 | 100%（亲眼看过） | 存疑（LLM 会编） |

**共同点：数据形态完全一致**——都是一棵挂着注释的"代码位置锚点"树。这正是插件已经拥有的核心资产。导航历史树和 LLM 生成的指引树可以共享同一个 UI 基座和锚点机制，只是节点来源不同。

两者还能叠加产生新价值：在导游图上标出"我实际走过并确认过的节点"，读者就能区分哪些理解是**验证过的**、哪些还只是 LLM 的**一面之词**。

### 4.3 插件的差异化定位：锚定层（Anchoring Layer）

"LLM 输出的文本和链接跟代码脱节"这个痛点，恰恰是本插件最应该占据的位置。

插件的差异化**不是和 LLM 比智能**，而是做锚定层：

- LLM 的每一句解释都钉在一个真实的、可点击跳转的代码位置上；
- 解释被组织成树，而不是一次性的对话文本；
- 跨会话持久，成为项目的"阅读地图"。

锚定同时顺手解决了信任问题：一个锚点点过去，代码和描述对不对得上，一眼可验。

### 4.4 三个需求逐个分析

**需求 1（找入口）**

- 可以给"入口"一个干净的定义：**控制流从项目外部进入项目内部的位置**。涵盖了 main 函数、CLI 命令、HTTP 路由、事件回调、扩展的 activate()、定时任务、消息队列消费者等各种"不能被用户简单执行"的入口。
- 大半是机械可查的（框架有约定，静态启发式可以找候选），LLM 负责分类和写一句话说明。
- 产出就是一组固定置顶的锚点——插件已有的 **pin 机制**天然承接。

**需求 2（看路径）**

- 离现有核心最近：插件已经在构建调用层级树了，只是目前只覆盖用户走过的地方。
- 扩展方向：
  - "在足迹之前展开"——VS Code 的 call hierarchy provider 本来就能静态展开；
  - LLM 给每个节点一行"这个子树提供什么"的摘要。
- 真正难的不是建树，而是**选择性展开**：大项目的完整调用树本身就是海量细节，与"略过细节"的目标冲突。需要重要性信号来决定展开哪些子树——已有的 **visit count** 就是一个现成信号，此外还有 LLM 判断、代码规模/扇出等。
- 其他已知难点：调用图实际是有环的图而非树（工具函数到处出现，需要噪声抑制）；动态分发、依赖注入、事件总线会打断静态调用链（LLM 可以补桥）。

**需求 3（理解实现）**

- 粒度与前两者不同：前两者的节点是**符号**，这个需要符号**内部的范围**（这段是 corner case 守卫、这段是主算法）。
- UI 形态：编辑器内的着色/标注（decorations），树上该函数节点下挂一层"段落"子节点。
- 它最依赖 LLM、结构上最难自动验证，但范围天然收敛在一个函数内，代价和风险都小。
- 注意锚点脆弱性：函数内部的范围锚点对编辑很敏感，需要基于符号而非裸行号来锚定（插件已有 tracableSymbol 的积累）。

### 4.5 范围警告与架构建议

**警告**：三个需求做全了就是一个 AI 代码阅读 IDE，和"简单且稳固"的初心是冲突的。LLM 带来的是另一类失败模式：延迟、成本、说错话。

**建议：分层架构，智能永远是可选的注释来源。**

- **Layer 0（内核，无 LLM 依赖，永远稳固）**：锚点 + 树 + 导航历史记录 + 可视化。就是现在的插件。
- **Layer 1（静态扩展）**：在足迹之前展开调用树；入口的静态启发式发现。
- **Layer 2（LLM 注释）**：入口说明、节点摘要、函数段落标注。写入同一棵树，缺了它内核照常工作。

**更进一步的路线**：插件可以**不自己调用 LLM**。2026 年每个人的编辑器里本来就有 agent——插件可以把"添加节点 / 给节点写注释 / 标注函数段落"暴露成 VS Code 命令或 MCP 工具，让外部 agent 把结构化的理解**写进**插件来。

这样：

- 智能的部分永远是别人的（模型升级免费享受，不背 API key、成本、延迟的包袱）；
- 插件只负责它最擅长的：**锚定、组织、可视化、持久化**；
- 既吃到 LLM 红利，又守住"简单且稳固"的初心。

## 五、现有工具盘点（2026-07 调研）

问题：有没有现成的"AI 阅读 IDE"类工具？结论：**这个赛道已经相当热闹，但几乎全部集中在"导游图"一侧；"锚定层 + 足迹"的位置仍然是空的。**

### 5.1 四类现有工具

**A. Repo → Wiki 生成器（编辑器外、网页形态）**——最热闹的一类：

- **DeepWiki**（Cognition/Devin）：任意 GitHub 仓库一键生成 wiki + 架构图 + 对话。
- **Google Code Wiki**（codewiki.google，2025-11 公开预览）：自动生成并持续维护交互式文档，架构/类/时序图，Gemini 对话；能从概念解释跳到具体代码文件。
- **Zread.ai**（智谱 Z.ai）：中文版 DeepWiki，仓库 → 结构化文档 + 架构图 + 对话；有 zread_cli 支持本地代码。
- 开源自部署替代：OpenDeepWiki、deepwiki-open、repowise、deepwiki-rs（C4 模型）、Code2Tutorial、PocketFlow Tutorial-Codebase-Knowledge 等。

**B. 编辑器内的可视化插件**——离本插件最近的一类：

- **CodeViz**（YC S24，VS Code 插件）：交互式代码地图，调用图（点节点跳代码、看上下游调用链）、AI 生成的 C4 架构图。
- **CodeVisualizer**：函数级流程图 + 模块依赖图。
- （史前项目：Sourcetrail，非 AI 的代码阅读可视化器，2021 年已停维护——说明纯静态可视化不足以撑起产品。）

**C. 面向 Agent 的代码理解基础设施**：

- **Driver AI**：用类编译器的方式预计算整个代码库的理解（符号级文档、架构图），通过 MCP 喂给 Cursor / Claude Code / Copilot。方向与本插件正好**相反**：它把上下文喂给 agent，本插件是让 agent 把结构写进可视化界面。

**D. 通用 AI 编程助手**（Cursor、Copilot、Claude Code、Augment、Greptile、Sourcegraph Cody 等）：都能"chat with codebase"，但输出是侧边栏文本——正是"和代码脱节"的原问题。

### 5.2 与三个需求的映射

- **需求 1（找入口）、需求 2（看路径）**：A 类工具在*文档层面*覆盖得不错——wiki 里有架构图、模块说明、指向代码的链接。但它们是**编辑器外的一次性产物**：链接单向跳入代码，跳进去之后工具就帮不上了；不知道你读过哪里、验证过哪里；代码一改就开始过期。
- **需求 3（理解实现）**：基本没人做锚定式的函数内部结构标注。现有方案都是"explain this"→ 侧边栏文本，脱节问题原样存在。
- **足迹 × 地图叠加**（4.2 节）：没有任何工具记录你自己的阅读轨迹并叠加在生成的地图上。所有现有工具都是纯自上而下的。

### 5.3 对本插件的含义

1. **4.5 节的警告被验证**：完整的"AI 阅读 IDE"已经有巨头（Google）、明星创业公司（Cognition、YC 系）在做，不要正面进入。
2. **差异化位置依然成立且无人占据**：编辑器内的锚定层 + 足迹与地图的对偶 + 让外部 agent 写入的开放接口，三者的组合没有现成对手。CodeViz 最接近，但它是纯导游图（无历史/足迹），且不开放给外部 agent 写入。
3. **A 类工具可以直接用**，与本插件互补而非竞争：读一个新项目时先开 DeepWiki/Zread 获得大方向（外层），进编辑器后由本插件承接精读过程（内层）。

## 六、聚焦 B 类路线：体验评估与编辑器能力盘点

### 6.1 对现有工具的试用结论（iwi）

- **A 类不是目标**：它一口气读完整个代码库生成理解，是静态的、一次性的；之后的所有交互都发生在 LLM 对话里（类似 NotebookLM 的设计理念），**远离了代码库**。
- **C、D 类也不是目标。**
- **B 类接近想要的形态**，但试用体感都相当一般：
  - **CodeViz**：交互式图形界面里动态请求 LLM 做结构分析——过程非常慢，体感不好。
  - **AtomicViz**：不走 LLM，直接用 Language Server 做符号解析和组织，体感明显好于 CodeViz。
  - **CodeVisualizer**：有文件级引用关系 + 函数内结构分析，但函数分析只是把每行代码 1:1 转成流程图，价值有限。

### 6.2 设计倾向：尽量不引入 LLM

引入 LLM 之后体验并不一定变好（慢、贵、不稳）。AtomicViz 的对比证明：**本地静态能力（Language Server / 符号解析）的即时响应本身就是体验优势**。这与 4.5 节的分层一致——优先把 Layer 0/1 做扎实，LLM（Layer 2）保持可选。

### 6.3 编辑器 UI 能力盘点（三问三答）

**Q1：能否动态改变代码窗口中的装饰（高亮一段、置灰其余）？——✅ 完全可以。**

- `window.createTextEditorDecorationType()` + `editor.setDecorations(type, ranges)`；支持 `backgroundColor`、`opacity`、`isWholeLine` 等。
- "聚光灯"手法：两个装饰类型，焦点外的 range 用 `opacity: '0.25'` 置灰，焦点段用高亮背景。
- 动态替换即时生效；`before`/`after` 伪元素可在行内注入文字（GitLens 行尾 blame 同款）——段落标签可直接渲染在段落首行行尾。
- 注意：装饰按 editor 实例应用，需监听 `onDidChangeVisibleTextEditors` 重挂；装饰本身不可点击。

**Q2：能否加自己的悬浮按钮（类似原生 quick fix）？——⚠️ 不能造自己的灯泡，但有三条正规渠道。**

1. **CodeLens**：行上方常驻可点击文字按钮（标准答案）。
2. **CodeActionProvider**：把自己的条目塞进原生灯泡 / Ctrl+. 菜单。
3. **Hover + trusted MarkdownString**：悬浮内容里放 `command:` 链接当按钮。
4. 进阶：**Comment API**（`comments.createCommentController`）——锚定到行的可展开行内富组件（GitHub PR 插件同款），是唯一原生的行内富 UI，适合"锚定的注释"。
- Hack 警告：装饰里画的图标不可点击，监听 selection 模拟点击的做法脆弱，不作主交互。

**Q3：能否弹出类似 definition/reference 的浮窗？——✅ 原生 peek 对插件开放。**

- `executeCommand('editor.action.peekLocations', uri, position, locations, 'peek')`：传入自己构造的 `Location[]`，弹出与 Peek References 一模一样的浮窗。
- Hover 浮窗仅限 Markdown（极小的 HTML 子集）；任意 HTML 的 webview **无法**锚定在代码位置悬浮（多年 feature request 未开放），富交互只能退到侧边栏 webview 或 comment thread。

**结论**：需求 3 的无 LLM 版本所需的 UI 原语已全部齐备——段落高亮 + 置灰（装饰）、段落标签（行内 `after` 文字）、段落列表（peek）、操作入口（CodeLens / hover 按钮），全本地、毫秒级。

### 6.4 更多可融入代码的原生 hook（第二轮盘点）

**嵌入代码文本流的：**

- **Inlay Hints**（`languages.registerInlayHintsProvider`）：真正嵌进文本流的行内小标签（参数名/类型提示同款），支持分段 label，每段可挂 tooltip、`location`（Ctrl+点击跳转）和 command——**可点击的行内标签**，比装饰的 `after` 文字更强。
- **Folding**（`languages.registerFoldingRangeProvider`）：自定义折叠区间 + `editor.fold` 系列命令程序化折叠——"略过海量细节"在编辑器里的原生机制：折掉 corner case 段落，只露主算法。
- **Semantic Tokens**（`registerDocumentSemanticTokensProvider` + `semanticTokenScopes` contribution）：自定义 token 类型参与语法着色，可按"语义角色"给代码换配色。
- **DiagnosticTag.Unnecessary** 技巧：诊断标 Unnecessary 的范围会原生**淡化显示**（未使用 import 同款）——另一种置灰手段，副作用是会进 Problems 面板。
- **DocumentHighlightProvider**：光标落在某处时自动高亮"相关"范围——"相关"由你定义（如：光标进入段落即点亮全段）。
- **Gutter 图标 + Overview Ruler**（装饰家族的独立表面）：行号侧图标、右侧滚动条彩条标记读过/标注过的位置。Minimap 不对插件开放。

**结构与导航骨架的：**

- **DocumentSymbolProvider**：一石三鸟——喂给 Outline 视图、**面包屑**、**Sticky Scroll**（`editor.stickyScroll.defaultModel` 默认走 outline）。把"段落"注入为符号，三处原生 UI 同时获得段落感知。
- **SelectionRangeProvider**：定义"扩选"（Shift+Alt+→）的层级——按段落逐级扩选。
- **自己当 Definition / Reference / CallHierarchy / TypeHierarchy provider**：不只消费这些能力，也可以为"虚拟锚点"提供跳转关系。
- **DocumentLinkProvider**：把任意文本范围变成 Ctrl+点击的链接（如把注释里的符号名变成可跳转）。

**编排与外联的：**

- **`revealRange` + selection 控制**：程序化滚动/定位，做"引导式阅读回放"。先例：**CodeTour** 插件——完全用原生 hook（Comment API 气泡 + 程序化导航）实现了代码导览，是"这条路能走通"的存在证明。
- **URI Handler**（`window.registerUriHandler`）：`vscode://publisher.ext/...` 深链——外部工具/文档/LLM 输出可以直接链到插件内的某个锚点。锚定层对外开放的关键通道。
- **Terminal Link Provider**：终端输出里自定义可点击链接——agent 在终端里提到的位置可被接住。
- **TextDocumentContentProvider + `vscode.diff`**：只读虚拟文档（可生成"地图页"）、程序化打开对比视图。
- **InlineValuesProvider**：调试时行内显示值，调试态的阅读辅助。

**明确不存在的**：ViewZones（在行间插入任意内容——只有 Comment API 和 CodeLens 能合法加行间内容）、minimap 着色、锚定在代码位置的任意 HTML 浮窗。

### 6.5 追问：能否折叠自定义的若干行？——✅ 可以，两条路

1. **FoldingRangeProvider**（正路）：返回任意 `FoldingRange(startLine, endLine)`，与语言自带的折叠区间合并；嵌套可以，**部分交叉的区间会被丢弃**；行级粒度。再用 `editor.fold` 命令（支持 `selectionLines` 参数）程序化折叠指定行处的区间。
2. **手动折叠区间命令**（免 provider）：先程序化设置 selection，再执行 `editor.createFoldingRangeFromSelection`（即 Ctrl+K Ctrl+, 的"手动折叠区间"），可对任意选区创建折叠；`editor.removeManualFoldingRanges` 移除。

边界：折叠**永远保留首行可见**（隐藏的是 startLine+1 到 endLine），无法整块完全隐藏；折叠占位文本不可自定义（collapsedText 未进稳定 API，只显示"…"）；用户若把 `editor.foldingStrategy` 设为 `indentation` 则 provider 被忽略；行中折叠（如折叠长字符串）官方不支持，Inline Fold 类插件靠装饰的 CSS 注入 hack 实现，脆弱不建议。

对需求 3 的用法："只看主算法" = 折叠所有次要段落，`editor.unfoldAll` 一键还原；首行可见的限制反而合适——段落首行 + 行尾标签正好构成"摘要行"。

（备忘：`vscode.lm` / Chat Participant API 可以走用户已有的 Copilot 订阅调模型、注册 `@navhistory` 聊天参与者——不需要自己管 API key。与 6.2 的倾向一致，仅作为将来 Layer 2 的备选记录。）

## 七、产品边界：加在原插件里，还是开新插件？

问题：任何一个"辅助代码理解"的新功能，直接加进 Navigation History（已有约 1100 下载），还是开新 Extension？

**判断：默认开新插件，用"承诺归属"作为每个功能的分界线。**

- **原插件的承诺**："记录你走过的地方，让你回得去。"顺着这个承诺生长的功能留在原插件：足迹树的静态扩展（先于足迹展开调用层级）、入口发现与置顶（给树提供更好的根节点）。
- **新承诺**："帮你读懂眼前这个函数"（需求 3 的阅读透镜：段落高亮/置灰、折叠摘要视图、段落进 outline）。与历史记录无功能耦合，属于新插件。

理由：

1. **风险隔离 = 4.5 节分层的产品化**。原插件是 Layer 0，价值在稳固；阅读辅助必然高频试错，churn 放在 0.x 新插件里是正常迭代，放在稳定插件里是差评。1100 个下载是真实用户，不该当实验田。
2. **市场定位**：“Navigation History”承载不了"代码阅读辅助"的搜索意图；新功能的实验性也不该污染老插件评分。
3. **先拆后合容易，先合后拆是灾难**。VS Code 有现成组合机制：`extensionDependencies`、`extension.exports` API、extension pack。将来"足迹 × 地图叠加"成立时，阅读插件可通过 API 读写 Navigation History 的锚点树，甚至最终合并。

成本提醒：solo 维护两个插件是双份发布/文档/issue 开销；共享代码（如 tracableSymbol 锚点逻辑）需抽库或复制。但需求 3 的首个实验（装饰 + 折叠 + DocumentSymbol）完全自包含、不依赖锚点树——**当下拆分的边际成本≈零**。

## 八、开放问题

1. **三个需求的优先级**：需求 2 离现有核心最近（增量扩展），需求 1 独立且性价比高，需求 3 最独立也最依赖 LLM。先做哪个？
2. **智能接入方式**：插件内置 LLM 调用，还是暴露命令/MCP 工具让外部 agent 写入？（4.5 倾向后者，待定。）
3. **阅读地图的持久化与分享**：一棵带足迹和注释的树，是否应该成为可以交给下一个读者（或下一个 agent）的项目级产物？存在哪里、什么格式？
4. **选择性展开的具体策略**：visit count、LLM 判断、扇出/规模，如何组合成一个"重要性"信号？
